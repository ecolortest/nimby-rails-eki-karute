# CLI廃止・完全GUI移行ガイド

このプロジェクトは現在 `nimby_passenger_tool.py` に業務ロジックとCLI入口が同居しています。
CLIを廃止してGUIへ完全移行する場合は、**「入口をGUIへ寄せる」より先に「ロジックの分離」を行う**のが安全です。

## 1. まず決めること（仕様）

- GUIで提供する画面一覧（路線、車両、ダイヤ、抽出、集計、DB確認）
- 既存CLIオプションのうち、GUIで必須なもの/不要なもの
- 既存JSON (`out/planning_db.json`) の互換性維持方針
- エラー表示方式（例外をGUIのダイアログで表示）

## 2. アーキテクチャ方針

推奨レイヤ:

1. **core**: 純粋な業務ロジック（I/O最小）
2. **services**: JSON入出力、動画抽出など副作用処理
3. **gui**: 画面、イベント、バリデーション、メッセージ表示

ポイント:

- `argparse` 依存コードをGUIから直接呼ばない
- GUIのボタンは「core/servicesの関数」を呼ぶだけにする
- 1機能1ユースケース（例: `add_vehicle_type(name, capacity)`）に分割する

## 3. 段階移行（推奨）

### Phase 1: ロジック抽出

- `nimby_passenger_tool.py` から、以下を順次関数化/モジュール化
  - DB操作（load/save、line/vehicle/timetable更新）
  - Ridership抽出・イベント化・集計
  - 曜日判定
- 既存CLIは薄いラッパーとして残す（この段階では廃止しない）

### Phase 2: GUI画面実装

- `startup_screen.py` は「起動画面」のみに限定
- メインGUI（例: タブ構成）を追加
  - 路線管理
  - 車両管理
  - ダイヤ管理
  - 抽出/イベント化/集計
  - 実行ログ/エラー表示

### Phase 3: CLI廃止準備

- GUIでCLI同等機能が揃ったら、READMEの主導線をGUIに変更
- CLIを非推奨として警告表示（1リリース程度）
- その後、CLIサブコマンドを削除

## 4. 最小実装チェックリスト

- [ ] 起動後にプロジェクトファイル選択/新規作成ができる
- [ ] 路線・車両・ダイヤのCRUDがGUIで完結する
- [ ] 抽出/イベント化/集計をGUIから実行できる
- [ ] 失敗時にダイアログで原因と再試行手順を表示する
- [ ] 既存データを破壊せず読み込める

## 5. CLI廃止時の注意

- 自動処理（CI・バッチ）でCLIが使われていないか事前確認する
- ドキュメントを「GUI中心」に全面更新する
- 廃止前に移行ガイド（旧CLI → 新GUIの対応表）を用意する

## 6. このリポジトリ向けの次アクション

1. `nimby_passenger_tool.py` の「コマンド解釈」と「処理本体」を分離する
2. `startup_screen.py` に遷移先（メイン画面）を追加する
3. READMEの利用手順をGUI起点に再構成する
